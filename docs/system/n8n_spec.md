## Architecture Decision: Reusable Template

**Date:** 2026-02-28 | **Status:** Confirmed — implement in current Airtable build

The n8n webhook flow connecting GHL → Airtable is built as a **reusable template**, not an operator-specific build. The operator name is passed as a **variable in the GHL webhook payload** (`{{operator}}`) and must never be hardcoded anywhere in the n8n logic.

Every new operator deployment automatically inherits the full data infrastructure with no new n8n build required.

**Per new operator, data infrastructure setup is:**
1. Add operator row to Airtable Operators table — 2 minutes
2. Copy GHL webhook workflow into new operator sub-account — 5 minutes
3. Update operator name variable in the GHL workflow payload — 2 minutes
4. Test with one conversation — 10 minutes

**Total: Under 20 minutes per new operator.** No new n8n build. No new freelancer hire. No new Airtable configuration.

**What must never happen:** The n8n flow must never have "Barley Bus" or any operator name hardcoded anywhere in the workflow logic. Verify by checking every node for hardcoded operator references before the freelancer contract is closed. The only place the operator name should appear is in the incoming webhook payload as `{{operator}}`.

**Impact on deployment velocity:** Data layer setup drops from a multi-hour build to under 20 minutes per operator. The bottleneck shifts entirely to GHL snapshot deployment and regression testing.

## Overview

After every Voice AI call, GHL sends a webhook to n8n. n8n processes the payload and writes a record to the Airtable Conversations table. Critical failure alerts are sent back into GHL via API Task + internal Note on the contact.

## n8n Flow Logic

### Configuration Node (Start of Flow)
A single Configuration node at the top of the flow holds the Airtable Base ID and any environment-level variables. All downstream nodes reference this node. Swapping bases or environments requires changing one node only.

### Step 1 — Receive Webhook + Sanitize
- Trigger: GHL webhook POST after every Voice AI call
- Validate payload: confirm `operator`, `contact_id`, and `call_date` are present. If missing, write to Errors table and stop.
- Sanitize `voiceai_summary` using a Function node immediately after the webhook trigger. Strip all non-ASCII characters: `summary.replace(/[^\x00-\x7F]/g, "")`
- Cast numeric fields to prevent silent logic drift from GHL sending numbers as strings:
  - `last_score = parseFloat(item.last_score) || 0`
  - `issue_count = parseInt(item.issue_count) || 0`
  - The `|| 0` handles both NaN and null — do not use a separate NaN check

### Step 2 — Operator Lookup
- Look up operator row in Airtable Operators table where `operator_name` matches payload `{{operator}}`
- Return: `operator_record_id`, `ghl_location_id`, `ghl_api_key`
- If no match found: write to Errors table, create GHL Task via default API key — "Unknown operator in webhook payload: {{operator}}" — and stop.

### Step 3 — Generate Event ID
- Construct `event_id` = `{{operator}}_{{contact_id}}_{{call_date}}`
- Standardize `call_date` to ISO-8601 format before concatenation — ensures the string is identical across webhook retries and prevents duplicate records from timestamp format variance

### Step 3.5 — Idempotency Check
- Search Conversations table for existing record where `event_id` matches
- If record exists: stop. Do not create a duplicate.
- This handles webhook retries and n8n retries cleanly.

### Step 4 — Critical Failure Check
Set `critical_failure = true` if any of the following:
- `last_score` < 2.0
- `issue_count` > 0
- `outcome` = `Escalated` AND `work_state` = `AI_ACTIVE` (escalation fired but state didn't flip — stuck state indicator)

Use an IF node with multiple outputs for the stuck-state branch.

### Step 5 — Write to Airtable
- Create new record in Conversations table
- Write `operator` as Linked Record using `operator_record_id` from Step 2 (not plain text)
- Map all payload fields per schema above
- Set `event_id` from Step 3, `critical_failure` from Step 4, `record_created_at` = now

### Step 6 — Critical Failure Alert
If `critical_failure = true`:
- Use HTTP Request node with `ghl_api_key` and `ghl_location_id` from Step 2 to call GHL API dynamically
- Create a Task assigned to Tim: `TOUROPS — Critical Failure — {{operator}} | Score: {{last_score}} | {{call_date}}`
- Add an internal Note to the contact record: `⚠️ Critical failure — Operator: {{operator}} | Contact: {{contact_id}} | Score: {{last_score}} | Outcome: {{outcome}} | {{call_date}}`

> **Why HTTP Request node, not GHL OAuth2 node:** The standard GHL OAuth2 credential in n8n binds to a single sub-account. Using the HTTP Request node with `ghl_api_key` pulled dynamically from Airtable per operator means one n8n template handles all operators without creating separate credential sets.

### Step 7 — Error Handling
- All steps wrapped in try/catch
- Any failure writes a record to the Airtable Errors table. Log only: `operator`, `contact_id`, `event_id`, `step`, `error_message`, first 500 characters of payload (excluding API keys or credentials)
- Never log `ghl_api_key` or raw full payload — keys fetched from Airtable must never appear in error records
- Silent failures are not acceptable — every error must surface in Airtable
- If Airtable returns a rate limit error (5 requests/second per base), enable batching or add a 200ms delay. Log rate limit errors to the Errors table like any other failure.

## Reusability Verification (Required Before Contract Close)

1. **No-hardcode audit:** Search every node in the completed n8n flow for any literal operator name. Expected result: 0 matches.
2. **Node naming:** All nodes use generic names (e.g., "Write to Conversations" — not operator-specific).
3. **Base ID check:** Airtable Base ID lives in the Configuration node only — not scattered across individual nodes.

## Per-Operator Setup (Once Flow Is Built)

2. Copy GHL webhook workflow into new operator sub-account — 5 min
3. Update operator name variable in GHL webhook payload — 2 min
4. Run one test conversation — verify record appears in Airtable Conversations table with correct operator linked record — 10 min

**Total: under 20 minutes per new operator.**
